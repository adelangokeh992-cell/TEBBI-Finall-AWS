# النسخ الاحتياطية — TEBBI

## التشفير

- محتوى النسخة الاحتياطية يُشفَّر قبل الحفظ في قاعدة البيانات (الحقل `data_encrypted` باستخدام مفتاح التطبيق من `ENCRYPTION_KEY` أو `JWT_SECRET`).
- النسخ القديمة التي تحتوي على الحقل `data` فقط (بدون تشفير) ما زالت قابلة للاستعادة؛ عند الاستعادة يُفضَّل الحقل `data_encrypted` إن وُجد.

## النسخ اليومية (جدولة)

لأتمتة إنشاء نسخة احتياطية يومياً:

1. **عبر API (مُوصى به):** استدعاء `POST /api/companies/{company_id}/backups` بانتظام (مثلاً مرة يومياً) باستخدام مستخدم لديه صلاحية (مثل `company_admin` أو خدمة ذات token ثابت).
2. **Cron مثال (Linux):** إنشاء مستخدم خدمة أو استخدام token:
   ```bash
   # يومياً الساعة 02:00 - استبدل BASE_URL و COMPANY_ID و TOKEN
   0 2 * * * curl -s -X POST -H "Authorization: Bearer YOUR_SERVICE_TOKEN" "https://your-domain/api/companies/COMPANY_ID/backups"
   ```
3. **سكربت في المشروع:** يمكن إضافة سكربت في `scripts/` أو `deploy/` يقرأ المتغيرات من البيئة ويستدعي الـ API ثم يصدّر الملف إلى مسار أو سحابة (انظر "موقع ثانٍ").

## تخزين النسخ في موقع ثانٍ

- بعد إنشاء النسخة عبر API، يمكن تصدير قائمة النسخ أو نسخة محددة إلى **موقع ثانٍ** (سحابة أخرى، خادم مختلف، S3، إلخ):
  - استدعاء `GET /api/companies/{company_id}/backups` ثم تنزيل بيانات نسخة معينة (يتطلب endpoint تصدير أو وصول مباشر إلى DB للتصدير).
  - أو تشغيل **mongodump** دوري على مجموعة `company_backups` (ومجموعات أخرى إن لزم) مع تشفير الملفات ورفعها إلى S3 أو خادم ثانٍ.
- يُنصح بتوثيق سياسة الاحتفاظ (retention) للنسخ في الموقع الثانٍ (مثلاً 30 يوماً، 6 أشهر، إلخ).

## اختبار الاستعادة

- **دورياً (شهرياً أو ربع سنوي):** في بيئة اختبار، استعادة من أحدث نسخة احتياطية والتأكد من أن الخدمة تعود والبيانات سليمة.
- توثيق تاريخ كل تجربة استعادة والنتيجة (نجاح/فشل) لأغراض الامتثال والمراجعة.
