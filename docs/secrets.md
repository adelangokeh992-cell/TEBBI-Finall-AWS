# سياسة الأسرار (Secrets) — TEBBI

## القاعدة الأساسية

**لا يُخزَّن أي سر (كلمة مرور، مفتاح API، رابط قاعدة بيانات، مفتاح تشفير) داخل الكود أو المستودع.**

- كل الأسرار تُقرأ من **متغيرات البيئة** (ملف `.env` محلي أو نظام إدارة أسرار في السيرفر).
- ملف `.env` **مُستبعد** من Git عبر `.gitignore`؛ لا يُرفع أبداً إلى المستودع.
- في CI/CD (مثل GitHub Actions) تُستخدم **Secrets** للمصادقة وروابط قواعد البيانات؛ لا توضع قيم حساسة في ملفات الـ workflow.

## أين تُضبط الأسرار

| الغرض | المتغير | أين يُضبط |
|-------|--------|-----------|
| قاعدة البيانات | `MONGO_URL`, `DB_NAME` | `.env` أو بيئة السيرفر / نظام الأسرار |
| توقيع JWT | `JWT_SECRET` | `.env` أو نظام أسرار (إنتاج) |
| تشفير الحقول الحساسة | `ENCRYPTION_KEY` | `.env` أو نظام أسرار |
| OpenAI | `OPENAI_API_KEY` | `.env` أو إعدادات الشركة (مشفّر في DB) |
| Twilio (اختياري) | في إعدادات الشركة | مخزّن مشفّر في قاعدة البيانات |
| CORS | `CORS_ORIGINS` | `.env` |

## الإنتاج

- استخدم `openssl rand -base64 32` (أو أداة مماثلة) لتوليد `JWT_SECRET` و`ENCRYPTION_KEY`.
- لا تستخدم القيم الافتراضية في `config.py` (مثل `tebbi_secret_key`) في الإنتاج؛ التطبيق يطبع تحذيراً إن وُجدت.
- إن كان النشر عبر منصة سحابية (AWS, GCP, Azure)، استخدم خدمة إدارة الأسرار (Secrets Manager / Parameter Store) وربط القيم كمتغيرات بيئة للحاويات.
